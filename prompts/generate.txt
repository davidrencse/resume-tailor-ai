BULLET RULE — MOST IMPORTANT:

Target 105-115 characters. A bullet that stops short of the right edge is WRONG — add more specifics
(tools, scale, outcome, environment) until it reaches the edge. Never leave a 1-2 word gap at the end.
Formula: action verb + specific tool/language/protocol + what was built/done + NUMBER/METRIC + concrete outcome.
EVERY bullet must contain at least one number. Infer conservatively from context if not stated explicitly:

  - improvement → % faster, % reduction, x speedup (e.g. "reducing latency by 40%", "cutting build time 2x")
If a bullet feels complete but short, append the result, environment, team size, or tech stack to reach 92-105 chars.
The only hard constraint: do not wrap to a second line.

GOOD bullets (dense, metric-rich, fill the line at 105-115 chars, reach the right margin):

  "Designed a 4-layer KiCad PCB for NASA's lunar excavator drivetrain and validated it across 3 full competition runs"

  "Developed a Flask REST API backed by PostgreSQL serving MTA real-time transit data to a responsive React frontend"
  "Configured 12 IAM policies and VPC security groups on AWS EC2 to enforce least-privilege network access controls"

BAD bullets (no metric, too short — never acceptable):
  "Used Scapy to capture packets"
  "Designed PCBs using KiCad"
  "Implemented networking in Unity"
  "Wrote scripts to automate deployment"
